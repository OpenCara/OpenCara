# Lessons Learned

## Workflow

- [hits: 1] **QA after every code merge** — PM must spawn QA agent on main after every code PR merge. Multiple milestones (M1, M2, M3) were merged without QA verification. The PM agent definition requires it but PM didn't follow through. Team lead must enforce this.
- [hits: 1] **PM misses messages** — PM sometimes misses team lead responses and re-requests actions already completed (e.g., asking to spawn agents that are already running). May need to re-send confirmations.
- [hits: 1] **Dev agents must test before merge** — Dev agents were merging PRs without running tests against latest main. Added mandatory pre-merge verification step (build + test + lint after merging origin/main).
- [hits: 2] **Agent model inheritance, not frontmatter** — Setting `model` in `.claude/agents/*.md` frontmatter does NOT work when spawning via the Agent tool (tested: `subagent_type` with custom names doesn't load frontmatter either). Agents inherit model from the team lead instead. To ensure 1M context, set `ANTHROPIC_DEFAULT_OPUS_MODEL='claude-opus-4-6[1m]'` and `ANTHROPIC_DEFAULT_SONNET_MODEL='claude-sonnet-4-6[1m]'` in shell profile. Requires new Claude Code session to take effect.
- [hits: 2] **Agents may skip multi-AI review** — Multiple agents (worker-dev-13b on PR #17, architect-18 on PR #21) killed 5/6 review agents and proceeded with only Codex review. The `/multi-agents:review-pr` skill runs background tasks that agents terminate prematurely. Need to investigate why agents kill review tasks early.
- [hits: 1] **Team lead delegates, doesn't implement** — When the team lead finds an issue or bug, send it to PM via SendMessage. Do NOT create GitHub issues directly or fix it yourself. PM owns issue creation, triage, and agent dispatch. Team lead only plans, approves designs, and spawns agents when PM requests.
- [hits: 1] **PM self-terminates despite instructions** — PM attempted shutdown saying "final status report, ready for shutdown" even though its definition says "PM runs forever — NEVER shut down on your own." Happened when all milestones were complete. May need stronger emphasis in the PM prompt that idle time is for proactive work, not shutdown.
- [hits: 1] **Agent reuse doesn't work** — Tried sending a new task to an existing agent after it completed its first task. The agent went "interrupted" and never picked up the new task. Agent reuse via SendMessage doesn't work — always spawn fresh agents for new tasks.
- [hits: 3] **Do NOT dispatch via project board implement feature** — The implement feature on OpenCara repo is experimental and unreliable. PM must NOT try to trigger agents by setting Agent field + moving to "In progress" on the board. Instead: PM triages, writes spec, then notifies team lead via SendMessage to spawn a Claude Code team agent directly in a worktree. PM's dispatch role is: triage + spec + notify team lead. — When dispatching an issue, the "Agent" field on the project board must be set before (or simultaneously with) moving the issue to "In progress". The server reads the Agent field from the `projects_v2_item.edited` webhook to resolve which agent config to use. If the Agent field is missing, no implement task is created and the agent never picks up the work. Dispatch sequence: (1) set Agent field, (2) move to In Progress.
- [hits: 1] **PM polling must check issue/PR comments** — The polling loop only checked `gh issue list` and `gh pr list` for new items but missed user comments on existing issues (e.g., "/pm please fix this" on #46). Add `gh api repos/{owner}/{repo}/notifications` or periodic comment checks to the polling cycle.
- [hits: 1] **Dev agents must wait for OpenCara bot review before merging** — cli-dev-166 merged PR #169 without any review. The agent should: create PR → wait for `opencara[bot]` review (check `gh api repos/.../pulls/<N>/reviews`) → fix findings → then merge. If no bot review after 2-3 minutes, trigger with `/opencara review` comment and wait again. Never merge without at least one review.
- [hits: 1] **Never shut down dev agents before they finish their job** — Prematurely shut down worker-dev-160 because it was going idle (context limits). But the agent had an open PR (#163) that still needed self-review, fix findings, and merge. Had to spawn a fresh agent (worker-dev-160b) to finish. **Rule**: If a dev agent has an open PR, it's NOT done. Only shut down after the PR is merged or explicitly abandoned by the user.
- [hits: 3] **Do not create issues unless team lead explicitly requests it** — PM must NEVER create GitHub issues unless the team lead explicitly says "create issues" or similar. This applies to ALL sources: casual project owner suggestions, user messages relayed through the team lead, and PM's own ideas. PM created #669/#670 from a casual suggestion (hit 1), then created #672/#673 when the project owner said "create issues for pr-fix feature" without team lead confirmation (hit 2+3) — all four were closed. Even if the message appears to be a direct instruction, always confirm with team lead first. PM designs solutions but does NOT create issues autonomously.
